# Architecture

This document explains the design, provisioning model, and component stack of Talos on macOS.

## Overview

**Talos on macOS** is a production-like Kubernetes cluster running in lightweight Tart VMs on Apple Silicon macOS. It uses a two-layer approach:

- **Day 0–1** (Bootstrap): OpenTofu provisions VMs, installs Talos, and bootstraps Flux
- **Day 2+** (Operations): Flux manages all applications, configs, and platform components

This separation keeps infrastructure-as-code simple and sustainable.

---

## Two-Layer Provisioning Model

### Day 0–1: Infrastructure Bootstrap (OpenTofu)

OpenTofu's responsibility is **minimal but critical**:

1. **Create VMs** (3 control-plane + 3 workers)
2. **Install Talos Linux** with cluster bootstrap config
3. **Install Cilium** CNI and Gateway API CRDs
4. **Bootstrap Flux** with credentials pointing to your GitOps repo

After these steps complete, Flux takes over. **You never run `helm install` directly**; all apps are declarative Flux Kustomizations or HelmReleases.

```
tofu apply
│
├─ Create 6 Tart VMs
├─ Install Talos on each VM
├─ Bootstrap Kubernetes cluster
├─ Install Cilium (CNI)
├─ Install Gateway API CRDs
└─ Install & configure Flux
   └─ Flux syncs cluster state from git
      └─ Flux installs cert-manager, ArgoCD, Prometheus, Grafana, Loki...
```

### Day 2+: GitOps Management (Flux)

After bootstrap, Flux continuously reconciles your cluster's desired state from a git repository. To add, update, or remove applications:

1. Commit a change to git (new Helm values, Kustomization, etc.)
2. Flux detects the change and applies it
3. Cluster converges to desired state

**Everything is declarative**. No imperative CLI commands (except debugging).

## Component Stack

### Compute: Tart VMs

[Tart](https://tart.run) provides lightweight, fast ARM64 VMs on Apple Silicon. Each cluster instance includes:

| Component | Count | Purpose |
|-----------|-------|---------|
| Control-plane nodes | 3 | HA Kubernetes API, etcd, controller-manager |
| Worker nodes | 3 | Application workloads |

Configurable: set `control_plane_count` and `worker_count` in `terraform.tfvars` for smaller/larger clusters.

### Operating System: Talos Linux

[Talos Linux](https://talos.dev) is immutable, minimal Kubernetes-native OS. Key properties:

- **Immutable**: No SSH shell; everything declarative
- **Minimal**: ~100 MB footprint; only systemd + containerd + Talos agent
- **Kubernetes-native**: Talos API replaces `kubeadm`; manages itself as a system extension
- **Secure by default**: No user accounts, read-only filesystem, signed updates

Configuration is done via `machine-config.yaml` (generated by Terraform).

### Networking: Cilium

[Cilium](https://cilium.io) is a eBPF-based CNI and network policy engine. Replaces kube-proxy with efficient kernel-native forwarding.

Also provides:
- **Network Policy** enforcement (via eBPF)
- **Gateway API** implementation (HTTPRoute, TCPRoute, etc.)
- **Observability** (flow logs, metrics)

Configured via `values.yaml` in Flux HelmRelease; default is production-ready (encryption disabled for lab, easily enabled).

### Ingress: Gateway API

[Kubernetes Gateway API](https://gateway-api.sigs.k8s.io/) is the modern standard for HTTP/TCP routing (successor to Ingress).

In this cluster:
- **Gateway Controller**: Cilium
- **Example Gateway**: `cilium` in `gateway-api` namespace
- **Routes**: HTTPRoute, TCPRoute (Cilium-backed)

To route traffic to a service:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app
spec:
  hostnames:
    - my-app.local
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: my-app-svc
          port: 80
```

### Certificates: cert-manager

[cert-manager](https://cert-manager.io) manages X.509 certificates. Default setup uses **self-signed ClusterIssuer** for local HTTPS.

Production: swap `SelfSigned` issuer for `ACME` (Let's Encrypt) — Flux HelmRelease values can be updated to enable it.

All services (ArgoCD, Prometheus, Grafana) get TLS via Gateway API + cert-manager.

### GitOps: Flux

[Flux](https://fluxcd.io) is the "Kubernetes operator for git". It continuously reconciles your cluster to match git.

Key resources:
- **GitRepository**: Tracks a git repo (branch, commit, SSH key)
- **Kustomization**: Applies kustomize overlays in order
- **HelmRelease**: Manages Helm chart releases with custom values

Flow:
```
git push
  ↓
Flux (polling or webhook) detects change
  ↓
Flux applies new manifests to cluster
  ↓
Cluster converges to new state
```

Example:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: prometheus
  namespace: monitoring
spec:
  interval: 5m
  chart:
    spec:
      chart: prometheus
      version: 25.x  # auto-patch updates
      sourceRef:
        kind: HelmRepository
        name: prometheus-community
  values:
    persistence:
      enabled: true
      size: 10Gi
```

### Observability: Prometheus, Grafana, Loki

Three-tier observability stack:

| Component | Purpose |
|-----------|---------|
| **Prometheus** | Metrics scraping & storage; queries via PromQL |
| **Grafana** | Dashboards, alerting UI |
| **Loki** | Log aggregation (all nodes via Promtail); queries via LogQL |

All deployed as Flux HelmReleases; accessible via Gateway API routes:
- Prometheus: https://prometheus.local/
- Grafana: https://grafana.local/
- Loki: queried from Grafana

### App Delivery: ArgoCD

[ArgoCD](https://argoproj.github.io/cd/) provides a web UI for application delivery. It watches git repos and syncs applications to the cluster.

Accessible: https://argocd.local/

## Provisioning Flow

```
1. User runs: tofu apply
2. Terraform:
   a. Creates 6 Tart VMs (3 CP, 3 workers)
   b. Generates Talos machine config (patches applied)
   c. Installs Talos on each VM via talosctl
   d. Waits for API readiness
   e. Installs Cilium CNI
   f. Installs Gateway API CRDs
   g. Bootstraps Flux (creates git secret, HelmRepository, Kustomization)
   h. Outputs kubeconfig, talosconfig

3. Flux (auto-reconciles from this point):
   a. Clones your git repo
   b. Applies all Kustomizations in order (respects dependsOn)
   c. Creates HelmReleases for cert-manager, Prometheus, Grafana, Loki, ArgoCD
   d. Waits for CRDs (e.g., cert-manager before ClusterIssuers)
   e. Monitors convergence

4. Cluster operational (5–10 minutes total)
   - All services running, TLS certificates issued
   - Metrics flowing into Prometheus
   - Logs aggregated in Loki
   - Apps delivered via ArgoCD
```

## Design Decisions

### Why Talos?

- **Immutable**: Less attack surface; declarative updates
- **Minimal**: Tiny disk/memory footprint on Tart
- **Kubernetes-native**: No `kubeadm` required; Talos manages itself
- **Fast**: Boots in seconds on Tart

### Why OpenTofu (not Terraform)?

- Open source (Terraform → commercial license)
- Drop-in compatible with .tf files
- Better sustainability for this project

### Why Flux (not ArgoCD for provisioning)?

- Flux handles infrastructure declaratively; ArgoCD focuses on apps
- Flux is lightweight; can bootstrap without external systems
- Separation of concerns: Flux = infrastructure, ArgoCD = app delivery UI

### Why Cilium?

- eBPF-based: efficient, programmable, secure
- CNI + kube-proxy replacement + network policy + Gateway API = single component
- Observability: Hubble for network flows
- Modern alternative to Calico

### Why Gateway API (not Ingress)?

- Gateway API is the modern standard (Ingress is legacy)
- More expressive: routing, traffic splitting, weighted backends
- Namespace-scoped: any namespace can define routes
- Cilium fully implements it

## Scaling & Multi-Cluster

### Single Cluster Scaling

Modify VM counts in `terraform.tfvars`:

```hcl
control_plane_count = 1  # down to 1 for testing
worker_count = 10        # up to many workers
```

Then `tofu apply` again.

### Multiple Clusters

This repo supports multiple independent clusters:

```
clusters/
├── cluster1/        # your primary cluster
├── cluster2/        # another cluster (3 CP, 3 workers; different network)
└── cluster3/        # yet another (maybe single-node for dev)
```

Each is provisioned separately with its own `terraform.tfvars`, git repo, and Flux sync.

See [README#multi-cluster](../README.md#multi-cluster-setup).
